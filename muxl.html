Media Uniform eXact Layout (MUXL)

date2026-04-17
editorsEli Mallon <eli@stream.place>
issueslist, new
abstract

MUXL defines a deterministic layout for media data in the ISO Base Media File Format (BMFF) for fMP4 files. Given the same encoded frames, any conformant MUXL implementation produces identical bytes, enabling stable content-addressed identifiers ([cid]) for video and audio content.

Introduction

Video is the most important media format on the web, but content-addressed systems don't know what to do with it. DASL gives us DRISL for deterministic serialization of structured data ([drisl]) and CIDs for content identifiers ([cid]). But MP4 files — the dominant container format for video — resist content-addressing. Run the same video through ffmpeg twice with identical settings and you'll get different bytes and no stable CID.

MUXL solves this by defining a single canonical container layout. Unnecessary metadata is stripped, box orderings are fixed, and all remaining fields are precisely specified. The result: given the same encoded frames with the same timestamps, any MUXL implementation produces the same bytes. Same bytes, same hash, same CID.

MUXL defines four representations of media data, each suited to a different use case:

  • MUXL fragments: the atomic unit — a single frame of video or a single sample of audio in a minimal moof+mdat pair. Suitable for low-latency streaming.
  • MUXL segments: one track's worth of fragments for one Group of Pictures (GoP). The canonical byte sequence over which content hashes and signatures are computed.
  • MUXL fMP4: a complete fMP4 file with an init segment followed by per-track segments. Suitable for storage, playback, and byte-range addressing.
  • MUXL flat MP4: a non-fragmented MP4 file with a complete moov at the start (faststart) and a single mdat. Suitable for editors, archival tools, and players that do not consume fMP4.

All four representations are deterministically interconvertible. Given the same encoded samples, any MUXL implementation produces byte-identical output for each representation. The content bytes — the encoded video and audio samples — are unchanged at every stage; only the container structure varies.

MUXL Fragments

A MUXL fragment is the smallest unit of media data in MUXL. Each fragment contains exactly one encoded sample (a single video frame or a single audio packet) wrapped in a moof+mdat pair.

A MUXL fragment consists of two BMFF boxes in sequence: 

[ moof ] [ mdat ]

moof Box

The moof box contains exactly one mfhd and exactly one traf:

  • mfhd: sequence_number is 1-based, incrementing per fragment within a track within a segment.
  • traf contains exactly three child boxes:
    • tfhd: track_id identifies the track. Flags must be 0x020000 (default-base-is-moof). No default sample values; all sample metadata is explicit in the trun.
    • tfdt: base_media_decode_time is the decode timestamp of this sample in the track's media timescale.
    • trun: exactly one sample entry. Flags: data_offset, sample_duration, sample_size, sample_flags. The sample_composition_time_offset flag must be set if and only if the sample has a non-zero composition time offset. data_offset is the byte offset from the start of the moof to the start of the sample data in mdat.

trun Sample Flags

Sample flags follow the BMFF sample_flags layout:

  • Sync sample (keyframe): 0x02000000. This sets sample_depends_on = 2, indicating the sample does not depend on any other sample.
  • Non-sync sample: 0x01010000. This sets sample_depends_on = 1 (depends on other samples) and sample_is_non_sync_sample = 1.

mdat Box

The mdat box contains exactly one sample's encoded data. Size is 8 bytes (4-byte big-endian size + 4-byte FourCC) plus the sample data length.

MUXL Segments

A MUXL segment is the canonical byte sequence over which content hashes are computed. Each segment contains one track's MUXL fragments for one GoP, concatenated in decode time order. 

Segment (track T, GoP N):
  moof(sample 1) + mdat(sample 1)
  moof(sample 2) + mdat(sample 2)
  ...
  moof(sample K) + mdat(sample K)

Each GoP with N tracks produces N segments — one per track. This per-track structure enables independent content hashing, byte-range addressing, and blind concatenation of segments for the same track.

Segmentation Rule

Segment boundaries are driven by video sync samples (keyframes). A new segment begins at each video keyframe. Audio samples are grouped with the video GoP they temporally overlap.

Given the same samples with the same timestamps, the segment boundaries are always identical. This property is what makes content-addressed hashing of segments stable.

For audio-only streams where there are no video keyframes, segmentation rules are not yet defined by this specification.

Content Addressing

Because MUXL's output bytes are deterministic, they are suitable as input to a content hash function. The same encoded frames always produce the same bytes, which always produce the same CID ([cid]).

Each track is hashed independently as a self-certifying segment (init segment + fragment data for that track and GoP). This supports subset verification (verify only the video track without needing the audio), track independence (drop or replace a track without invalidating others), and multi-track streams with multiple synced video and audio tracks.

The hash algorithm and signing mechanism are not specified by MUXL. MUXL defines what bytes are canonical; the signing layer (such as S2PA) defines how to sign them.

Self-Certifying Format

A self-certifying MUXL segment is a MUXL segment prepended with its corresponding init segment. This produces a standalone, playable fMP4 file containing exactly one GoP's data for one track, together with the track configuration needed to decode it. 

Self-Certifying Segment (track T, GoP N):
  ftyp + moov (init segment for track T)
  moof(sample 1) + mdat(sample 1)
  moof(sample 2) + mdat(sample 2)
  ...
  moof(sample K) + mdat(sample K)

The content hash for a self-certifying segment is computed over the entire byte sequence — init segment and fragment data together. Including the init segment in the hash input prevents an attacker from substituting a different init segment (e.g., altered codec parameters, modified SPS/PPS, changed timescale) that would cause the same encoded samples to decode differently. Because the self-certifying format is a logical construct used for verification rather than a format stored on disk or transmitted over the wire, the small redundancy of hashing the init bytes is negligible.

A verifier with access to a self-certifying segment can:

  1. Hash the complete self-certifying segment bytes (init segment + fragment data) to obtain a CID.
  2. Verify the CID against a signed manifest or other trust anchor.
  3. Play back the content using the included init segment for codec configuration.

The init segment in the self-certifying format contains only the single track that the segment belongs to, not all tracks in the stream. This is a per-track init segment as described in the Init Segment section, constructed with a single trak box.

MUXL fMP4

A MUXL fMP4 is a valid fMP4 file consisting of an init segment followed by per-track segments grouped by track: 

ftyp
moov (init — track config, empty sample tables)
[track 1 segments: GoP 1, GoP 2, ...]
[track 2 segments: GoP 1, GoP 2, ...]
...

Each track's segments form a contiguous byte range within the file. Tracks are ordered by track_id ascending. This layout enables HLS byte-range playlists to reference individual tracks within a single file.

A MUXL fMP4 is a standard fMP4 file playable by any fMP4-capable player. The init segment at the start provides all codec configuration needed for playback, and each track's contiguous segment range makes HLS byte-range playlists straightforward.

fMP4 files are appendable: as new GoPs arrive (e.g., during a livestream), their segments can be appended to each track's byte range. The init segment is stable as long as the track configuration does not change.

MUXL Flat MP4

A MUXL flat MP4 is a single file that serves two roles simultaneously:

  1. To parsers that inspect top-level boxes, it is a flat (non-fragmented) MP4 with a faststart moov and a single mdat. Video editors, desktop players, and any tool that handles classic progressive MP4 treats it as an ordinary file.
  2. To HLS byte-range CMAF consumers, the same bytes contain a sequence of self-contained moof+mdat fragments addressable by byte range. Those fragments are the MUXL fMP4's body bytes, relocated verbatim.

The trick: the mdat at the top level is an envelope. Its payload is the concatenation of per-sample moof+mdat pairs, grouped by track. A flat-MP4 parser sees only the outer mdat and reads sample bytes via co64 offsets that point past the inner moof headers. An HLS player fetching a byte range gets a self-contained CMAF fragment and never knows the envelope exists. This envelope pattern is adapted from OBS Studio's hybrid MP4 design ([obs-hybrid-mp4]). 

ftyp
moov (populated sample tables; no mvex; faststart)
mdat (64-bit largesize envelope; payload =)
  [track 1: moof+mdat, moof+mdat, ... ]    ← per-sample CMAF fragments
  [track 2: moof+mdat, moof+mdat, ... ]
  ...

Because the moov precedes the mdat and carries absolute file offsets in co64, writing a MUXL flat MP4 requires a metadata pre-pass: each inner moof's encoded size must be known before per-sample file offsets can be computed. The moov's encoded size is invariant under co64 value changes (entries are fixed-size 64-bit integers), so the encoded size converges after a single placeholder encode.

Relationship to the MUXL fMP4

The inner moof+mdat sequence of a MUXL flat MP4 is byte-identical to the body of a MUXL fMP4 produced from the same samples. Converting an fMP4 into a flat MP4 is therefore a metadata-only rewrite:

  1. Parse each fMP4 moof to recover per-sample metadata (duration, size, sync flag, composition offset). No sample-data reading is required.
  2. Build the populated flat-MP4 moov.
  3. Replace the fMP4's init moov with the flat moov, prepend the 16-byte outer mdat envelope header, and copy the fMP4's moof+mdat body verbatim.

The inverse (flat MP4 → fMP4) is similarly a wrapper swap: drop the outer envelope header, replace the populated moov with an init moov, and leave the inner fragments untouched.

moov

Reuses the canonical mvhd, trak, tkhd, mdhd, hdlr, and minf rules from the Init Segment section, with:

  • Populated stbl sample tables (see next section).
  • No mvex. The top-level view is non-fragmented; HLS consumers use a separate init segment delivered out of band.
  • mvhd.duration: maximum per-track duration, rescaled to the movie timescale 1000.
  • tkhd.duration: the track's total decode duration in the movie timescale.
  • mdhd.duration: the track's total decode duration in the media timescale.

stbl

Each track's stbl carries populated sample tables. co64 entries point at sample bytes inside the inner mdats, not at the start of each inner fragment.

  • stsd: same as the Init Segment form.
  • stts: run-length encoded per-sample decode durations in the media timescale.
  • ctts: present iff any sample has a non-zero composition time offset; version 1 (signed 32-bit offsets); run-length encoded.
  • stsz: uniform form when all samples are the same size, otherwise per-sample list.
  • stsc: exactly one entry — first_chunk = 1, samples_per_chunk = 1, sample_description_index = 1. Each sample sits in its own chunk because each is preceded by its own inner moof+mdat header bytes.
  • co64: one entry per sample. Entry i is the absolute file offset of sample i's bytes within its inner mdat (inner_moof_start + inner_moof_size + 8). co64 is always used, never stco, for deterministic moov sizing.
  • stss: 1-based sync sample indices, for video tracks only. Omitted for audio tracks and for any track in which every sample is a sync sample.

No other sample table boxes (stsh, stps, stdp, padb, sdtp) are emitted.

Outer mdat

The outer mdat header uses the 64-bit extended size form (size = 1, followed by an 8-byte largesize), always — regardless of total payload — so that its header is always 16 bytes and the layout arithmetic is independent of file size.

The outer mdat payload is the concatenation of inner moof+mdat pairs, grouped by track_id ascending, with samples within a track in decode order.

Inner moof+mdat fragments

Each inner fragment is a canonical MUXL fragment as defined in the MUXL Fragments section: a minimal single-sample moof+mdat pair with default-base-is-moof and tfdt. These fragments are byte-identical whether they appear in a MUXL fMP4 body or inside a MUXL flat MP4 envelope.

The mfhd.sequence_number increments globally across all tracks, starting at 1 for the first fragment in the file.

Init Segment

The MUXL init segment consists of an ftyp box followed by a moov box. It contains track configuration with empty sample tables, zero durations, and no sample data. The init segment is used in MUXL fMP4 files (followed by per-track segments) and in self-certifying segments (followed by a single track's segment data). The flat MP4 representation reuses all of the init segment's box structure rules except for the sample tables, which it populates (§ MUXL Flat MP4 → stbl), and mvex, which it omits.

ftyp

The ftyp box is identical across all MUXL representations that carry one (MUXL fMP4 and MUXL flat MP4):

  • major_brand: muxl
  • minor_version: 0
  • compatible_brands: [muxl, isom, iso2]

The muxl major brand signals that the file conforms to this specification. The isom and iso2 compatible brands keep the file playable by generic ISOBMFF tools that do not recognize muxl. The ftyp is codec-agnostic; players detect the codec from stsd entries in the moov.

moov

The moov box contains exactly: mvhd, one or more trak boxes (sorted by track_id ascending), and mvex.

The following boxes must not be present at the moov level: udta (non-deterministic tool tags), meta, iods, and free/skip.

mvhd

Canonical mvhd values:

  • version: 0
  • flags: 0
  • creation_time: 0
  • modification_time: 0
  • timescale: 1000
  • duration: 0
  • rate: 1.0 (0x00010000)
  • volume: 1.0 (0x0100)
  • matrix: identity matrix
  • next_track_id: max(track_ids) + 1

mvex

The mvex box signals that moof+mdat pairs follow the moov. Required for fMP4 playback. Contains one trex per track:

  • track_id: matching the corresponding trak
  • default_sample_description_index: 1
  • default_sample_duration: 0
  • default_sample_size: 0
  • default_sample_flags: 0

All sample metadata is explicit in each trun entry, so trex defaults are all zero.

trak

One trak per track, sorted by track_id ascending. Each contains tkhd and mdia. No udta, meta, or free/skip.

tkhd:

  • version: 0
  • flags: 3 (track_enabled | track_in_movie)
  • creation_time: 0
  • modification_time: 0
  • track_id: the track's identifier
  • duration: 0
  • width, height: video dimensions for video tracks; 0 for audio tracks
  • volume: 0 for video tracks, 1.0 (0x0100) for audio tracks
  • matrix: from track configuration
  • layer: 0
  • alternate_group: 0

mdia contains mdhd, hdlr, and minf:

mdhd:

  • version: 0
  • flags: 0
  • creation_time: 0
  • modification_time: 0
  • timescale: preserved from source (passthrough to maintain timestamp precision)
  • duration: 0
  • language: "und" (undetermined)

hdlr:

  • version: 0
  • flags: 0
  • handler_type: "vide" for video, "soun" for audio
  • name: empty string (cosmetic, varies across muxers)

minf:

  • vmhd: present for video tracks (default values).
  • smhd: present for audio tracks (default values).
  • dinf: contains dref with one url entry (empty location, flags = 1 / self_contained).
  • stbl: stsd populated with the track's codec configuration. All other tables (stts, stsc, stsz, stco) present but empty, since the init segment contains no samples.
  • edts (optional): when present in the source track, preserved byte-for-byte on output. Contains a single elst whose entries use segment_duration in the movie timescale (1000) and media_time in the track's media timescale, with -1 denoting an empty edit. Edit lists are track-level presentation metadata, not per-fragment metadata; a fragments-only round-trip drops them, while a fragments + catalog round-trip preserves them.

Codec Configuration

Each track's stsd contains a codec-specific sample entry. Currently supported:

  • H.264/AVC: avc1 + avcC (SPS/PPS)
  • AV1: av01 + av1C
  • Opus: Opus + dOps
  • AAC: mp4a + esds

Codec configuration bytes are preserved from the source without modification. MUXL does not normalize codec-level parameters.

Constructing MUXL from fMP4

The steps to construct MUXL fragments from an fMP4 stream are:

  1. Accept a byte stream input containing a valid fMP4 file (an init segment followed by moof+mdat pairs).
  2. Read the ftyp and moov boxes from input to obtain the track configuration. Store the track configuration in catalog.
  3. For each track, initialize a decode_time counter to 0 and a sequence_number counter to 0.
  4. For each moof+mdat pair in input:
    1. Parse the moof to obtain track ID, sample durations, sizes, flags, and composition time offsets. Resolve defaults via the BMFF cascade: trun > tfhd > trex.
    2. Read the corresponding mdat payload.
    3. For each sample in the trun:
      1. Increment sequence_number for the sample's track.
      2. If the moof contains a tfdt, use its base_media_decode_time as decode_time for the first sample and advance by each sample's duration for subsequent samples. Otherwise, continue from the previous decode_time.
      3. Write a MUXL fragment: a new moof with a single-sample trun using canonical field values (as specified in the MUXL Fragments section), followed by an mdat containing the sample's encoded data.
  5. Return catalog and the sequence of MUXL fragments.

The steps to construct MUXL segments from MUXL fragments are:

  1. Accept a sequence of MUXL fragments and a catalog describing the tracks.
  2. Identify which track IDs are video tracks using catalog.
  3. Maintain per-track buffers, keyed by track ID.
  4. For each fragment:
    1. If the fragment belongs to a video track and is a sync sample (keyframe), and at least one keyframe has been seen before, then the current GoP is complete. Emit a MUXL segment for each non-empty per-track buffer and clear the buffers.
    2. Append the fragment's bytes to the buffer for its track ID.
  5. After all fragments have been processed, emit a final MUXL segment for each non-empty per-track buffer.

The steps to construct a MUXL init segment from a catalog are:

  1. Accept a catalog containing video and audio track configurations (codec, dimensions, sample rate, channels, track ID, and timescale for each track).
  2. Write an ftyp box with the canonical values specified in the Init Segment section.
  3. Collect all tracks from catalog and sort by track_id ascending.
  4. For each track, construct a trak box with the canonical field values specified in the Init Segment section, using the track's codec configuration for the stsd entry.
  5. Construct a moov box containing mvhd, mvex (with one trex per track), and the sorted trak boxes.
  6. Return the concatenation of the ftyp and moov boxes.

The steps to construct a MUXL fMP4 are:

  1. Accept a catalog and a sequence of MUXL segments, grouped by GoP.
  2. Construct a MUXL init segment from catalog.
  3. Collect all track IDs and sort ascending.
  4. For each track ID in sorted order, concatenate all of that track's segments (GoP 1, GoP 2, ...) in order.
  5. Return the init segment followed by the per-track concatenated segments.

The steps to construct a MUXL flat MP4 are:

  1. Accept a catalog and, for each track, the ordered sequence of the track's samples with per-sample metadata (duration, size, sync flag, composition time offset) and a reference to each sample's encoded bytes.
  2. Assign a global 1-based mfhd.sequence_number to each sample in the order "all of track k's samples before track k+1's, tracks in track_id ascending order, samples within a track in decode order." For each sample, build the inner moof as a canonical MUXL fragment moof and measure its encoded size mi. Record the inner fragment size fi = mi + 8 + sizei.
  3. Build each track's canonical stbl (stts, ctts if needed, stsz, stsc, co64, stss if needed) with co64 filled with the correct number of placeholder entries (one per sample).
  4. Construct the moov using the canonical mvhd, trak, and mdia rules, with track and movie durations filled in. Do not include mvex.
  5. Construct the ftyp with canonical values. Measure the sizes F of the encoded ftyp and M of the placeholder moov.
  6. Compute the outer mdat payload start offset P = F + M + 16 (the outer mdat header is an 8-byte size/type field plus an 8-byte largesize).
  7. For each sample i in the same global order as step 2, compute its absolute file offset: P plus the sum of all preceding inner fragment sizes f, plus mi + 8 (past this sample's inner moof and inner mdat header). Install the result in the sample's co64 entry.
  8. Re-encode the moov with the finalized co64 values. Its byte length is unchanged from the placeholder encode, because co64 entry count is the same and each entry is a fixed-size 64-bit integer.
  9. Write the ftyp, the re-encoded moov, and the outer mdat header (size field 0x00000001, type mdat, largesize = 16 + Σ fi).
  10. For each sample i in the same global order, write the canonical moof (with trun.data_offset = mi + 8) followed by the inner mdat containing the sample's encoded bytes.

The steps to convert a MUXL fMP4 to a MUXL flat MP4 are:

  1. Parse the fMP4's moof boxes to recover per-sample metadata. No sample-data reading is required.
  2. Construct a new flat-MP4 moov from that metadata, sized as above.
  3. Write the fMP4's ftyp unchanged, then the new moov, then the 16-byte outer mdat envelope header, then stream the fMP4's moof+mdat body verbatim.

Determinism

MUXL's value proposition depends on determinism: the same input must always produce the same output. The following properties are guaranteed:

  • Init segment determinism: given the same track configuration (codecs, dimensions, sample rates, timescales, track IDs), any MUXL implementation produces byte-identical init segments.
  • Fragment determinism: given the same encoded sample data, track ID, decode timestamp, duration, sync flag, and composition time offset, any MUXL implementation produces byte-identical fragments.
  • Segment determinism: given the same sequence of encoded samples with the same timestamps and sync flags, any MUXL implementation produces byte-identical segments with identical segment boundaries.
  • Flat MP4 determinism: given the same track configuration and the same ordered per-track sequence of encoded samples (with their durations, sync flags, and composition time offsets), any MUXL implementation produces a byte-identical flat MP4. The inner moof+mdat body is byte-identical to the corresponding MUXL fMP4 body. The co64 offsets are deterministic because the moov's encoded size is a pure function of sample metadata, not sample offsets.

The following are not currently normalized by MUXL and therefore represent sources of non-determinism across different source encoders:

  • Media timescale: the mdhd timescale is passed through from the source. Different encoders may use different timescales (e.g., 16000 vs. 60000 for 60fps video), producing different duration values in trun entries.
  • Audio priming samples: muxers disagree on how to handle encoder delay for Opus and AAC. Some keep priming samples in the media data with an edit list offset; others strip them.

These sources of non-determinism are acceptable for the primary use case of livestream ingest from a single source encoder. Future revisions of this specification may address cross-encoder canonicalization.

Additional Considerations

Relationship to CMAF

The MUXL fMP4 is a constrained profile of CMAF. It uses single-sample fragments (one frame per moof+mdat pair), the most granular CMAF chunking mode. Any CMAF-compatible player can play a MUXL fMP4.

The MUXL flat MP4 is not a CMAF file at the top level — its top-level boxes are ftyp+moov+mdat with no mvex. But its outer mdat payload is a concatenation of CMAF fragments, so HLS byte-range playlists can reference those fragments directly. An HLS player never parses the outer container; it only receives CMAF-compliant byte ranges.

Relationship to Hang / Media over QUIC

MUXL fragments are directly compatible with Hang CMAF ([hang]), the container format used in Media over QUIC. Hang transmits per-frame moof+mdat pairs over QUIC with codec config out-of-band. MUXL can consume Hang frames directly and canonicalize them into segments.

Security & Verifiability

MUXL provides byte-level determinism for content addressing but does not define a signing or verification mechanism. For provenance verification, use a signing layer such as S2PA that computes per-track hashes over self-certifying segments and publishes signed manifests.

The self-certifying format (init segment + one track's segment data) provides a minimal verifiable unit: it can be independently played back, hashed, and checked against a signed manifest without access to the full fMP4 or stream.

Conformance

A conformant MUXL implementation must produce byte-identical output for the same input:

  • All box orderings must follow this specification exactly.
  • All canonical field values (timestamps, flags, metadata) must match this specification exactly.
  • Stripped boxes (udta, meta, free, skip, iods) must not be present.
  • Segment boundaries must occur at the same positions (video keyframe boundaries) for the same input.

Appendix: Media Type

MUXL fMP4 files and MUXL flat MP4s are valid MP4 files. The appropriate media type is video/mp4 (or audio/mp4 for audio-only files). The conventional file extension is .mp4.

MUXL segments (without an init segment) use the extension .m4s, following the CMAF convention for media segments.
